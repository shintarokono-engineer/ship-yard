「何を、どの順序で、どこまで」を実装前に決めます。実装は別途 `/loop` や手動で進める。

## いつ使うか

- 新機能、API 変更、リファクタを含む実装、複数ファイル横断などの変更の前。
- 影響範囲が見えづらい変更（DB スキーマ変更、共通基盤改修など）の前。
- **使わない**: 1 ファイル完結の typo 修正・小バグ修正（即実装でよい）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **対象タスク**: チケット ID / `understanding-ticket` の出力 / 自由記述
   - **スコープ**: 最小限 / 関連リファクタも含む / フェーズ分割で段階リリース
   - **制約**: 期限 / 並行作業者 / リリース窓 / 機能フラグの有無
   - **既知の影響範囲**: 自分が把握済み / 探索が必要（→ `exploring-related-code` を先に）
   - **リスク許容度**: 後方互換絶対 / 破壊的変更可だが事前告知必須 / 内部のみで影響限定
   - **テスト戦略**: ユニットのみ / 統合追加 / e2e 追加 / characterization テスト先行

2. **計画を組み立て**:
   - 影響範囲を列挙（ファイル / モジュール / 外部接点）
   - Phase 分け（最小から段階的に）
   - 各 Phase の完了条件 = テストが通ること + レビュー観点
   - リスクと対策（rollback 手順含む）
   - 並行不可な作業の順序を明示

3. **代替案も検討**:
   - 「やらない」選択肢の検討
   - 既存ライブラリ/機能で代替できないか
   - スコープを縮小できないか

4. **成果物を `.claude/output/planning-implementation/<YYYY-MM-DD-HHmm>-<topic>.md` に書き出し** — 標準出力には Phase サマリのみ。

## 出力フォーマット

```markdown
# 実装計画: <タイトル>

## 目的

<1〜2 行: なぜやるか>

## ゴール（受け入れ条件）

- [ ] ...
- [ ] ...

## スコープ

**含む**:

- ...
  **含まない**:
- ...

## 影響範囲

- フロント: <ファイル/領域>
- バック: <ファイル/領域>
- DB / インフラ: <あれば>
- 外部接点: <API / 他チームの依存先>

## Phase

### Phase 1: <名前>

**目的**: <この Phase で何が達成されるか>
**手順**:

1. ...
2. ...
   **完了条件**:

- テスト: <該当テストが緑>
- レビュー観点: <レビュアーが確認すべき点>

### Phase 2: ...

## リスクと対策

| リスク | 影響 | 対策 / Rollback |
| ------ | ---- | --------------- |
| ...    | ...  | ...             |

## 代替案（検討して却下したもの）

- **案 A**: <内容> — 却下理由: <理由>

## 次のステップ

- Phase 1 着手 → 完了したら本計画を更新
- 不明点が残るなら `understanding-ticket` に戻る
```

## ルール

- **自明な将来要件のための抽象化を計画に入れない**。今必要なものだけ。
- **Phase は最小から**。最初の Phase で価値が出ること（部分リリース可能）。
- **rollback を必ず書く**。後戻りできない変更（DB マイグレーション）は特に。
- **コードを書かない**。計画段階。実装は別。
- 不明点が残るなら計画にせず、`understanding-ticket` に戻ってヒアリング継続。
