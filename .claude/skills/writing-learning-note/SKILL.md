作業・調査で得た技術知識を、既存の学習ノート vault に「あとで読み返せる単一トピックのメモ」として落とし込みます。vault は本人が長期蓄積している資産なので、**既存の分類・構成・文体に合わせること**と、**書く前に内容を提案して承認を取ること**を最優先にします。

## vault の場所と分類

- ルート: `/Users/s.k/document/学習ノート/`
- 分類はトップレベルのフォルダ = カテゴリ。1 ノート = 1 ファイル = 1 トピック。

| フォルダ | 扱うトピック |
| --- | --- |
| `開発環境/` | モノレポ、デプロイ、バージョン管理ツール 等の足回り |
| `バックエンド/` | NestJS、Stripe、メール送信、Anthropic API、Embedding/RAG 等 |
| `データベース/` | PostgreSQL、Prisma 等 |
| `Web開発/` | フロントエンド、React、Next.js、Node.js、TypeScript、認証、SEO 等 |

| `C#/` `ITパスポート/` | 言語・資格学習(構成が異なるので原則触らない) |

着手時に必ず実際のファイル一覧を取得して、最新の分類・既存ノートを確認します:

```bash
find "/Users/s.k/document/学習ノート" -name "*.md" -not -path "*/.obsidian/*" | sort
```

## ワークフロー

### 1. インプットの種類を見分ける

- **トピック指定**(「TypeScript の学習ノートに readonly 配列を足して」): 対象トピックが明確。該当ノートの有無を確認する。
- **作業ベース**(「今日の ship-yard の作業を学習ノート化して」「これ学んだこと記録して」): 直近の会話・実装・diff から「再利用可能な技術知識」を抽出する。プロジェクト固有の仕様・進捗ではなく、**他プロジェクトでも通用する一般知識**だけをノート化対象にする(例: 「ship-yard の F10 で SEO 実装した」ではなく「Next.js App Router の SEO 実装パターン」)。

### 2. 既存ノートを確認し、新規 or 追記を決める

- 該当トピックのノートが既にある → **追記**(後述の追記ルール)。書く前に既存ノートを Read し、**同じ内容が無いか**確認する。重複は足さない。
- 該当ノートが無い → **新規作成**。適切なカテゴリフォルダに `<トピック>.md` を作る。

### 3. 提案して承認を取る(書く前に必ず)

実ファイルを書き換える前に、以下を簡潔に提示してユーザーの承認を得ます:

- 対象ファイル(新規 or 追記、フルパス)
- 新規なら: タイトルと章立て(`## N.` 見出しの一覧)
- 追記なら: どのセクションに、どんな見出しで、何を足すか

承認なしに vault のファイルを作成・編集しないこと。複数ノートにまたがる提案なら、箇条書きで全体像を出してから個別に進めます。

### 4. 執筆する

`references/note-format.md` のフォーマット規約とテンプレートに従って書きます。文体・構成がブレると vault の一貫性が崩れるため、書く前に必ずこの reference を Read してください。

**端的に、必要な情報だけ書く。** これは教科書ではなく本人があとで素早く思い出すための個人メモ。網羅性より「再利用できる核」を優先する。用途の列挙・ハマりどころ・アンチパターン・追加の例は、**それぞれが本当に価値を足す時だけ**入れる。「一通り埋める」発想で表やセクションを増やさない。迷ったら削る。既存 SEO ノートのような大作は、トピックが広い時の結果であって、目指す型ではない。

**粒度を対象で使い分ける**:

- **概念・仕組み・全体像・なぜ(why)**: 理解に必要なだけ説明してよい(現状の深さで OK)。図・段階の説明・トレードオフはここで効く。
- **個別の API(メソッド / 演算子 / 関数 / フック / オプション)**: 「**それが何で、何ができるか**」が分かれば十分。1〜数行で要点を書き、例は必要な時に**小さいものを 1 つ**だけ。比較表・複数の用途・網羅的なコード例で膨らませない。1 つの API に何十行も使わない(目安は数行〜十数行)。ユーザーが明示的に「○○との違い」を求めた場合だけ、その比較を最小限で添える。

### 5. 書いた後

- 何を追加/作成したかを 1〜2 行で要約して報告する。
- vault は git 管理下だが、**コミットはしない**(本人が Obsidian 側で管理)。コミットを頼まれたら明示指示として扱う。

## 重要な制約(短縮版・詳細は reference)

- **文体はだ・である調 + 体言止めの簡潔なメモ調**。ですます調にしない。ship-yard の CLAUDE.md は「ですます調」だが、それはコード/ドキュメント用であって**この vault には適用しない**。既存ノートの文体が SSoT。
- **プロジェクト固有情報・秘密情報を残さない**。リポジトリ名・社名・APIキー・個別の進捗は書かず、汎用化した知識として書く。**コード例の中の固有名も一般化する**(自前ヘルパー名・関数名・URL・変数名は `getSiteUrl()` → `getBaseUrl()` のような汎用名や `example.com` 等のプレースホルダに置き換える)。
- **既存の番号体系を壊さない**。追記でセクションを挿入する時は連番のズレに注意する。
