# 学習ノート フォーマット規約

vault の既存ノート(`Web開発/SEO.md`、`バックエンド/Stripe.md`、`Web開発/TypeScript.md` など)から抽出した、守るべき構成・文体ルール。新規・追記いずれもこれに合わせる。

## 大前提: 端的に書く

教科書ではなく**本人の想起用メモ**。網羅性ではなく「再利用できる核」を最短で示すのが価値。用途の列挙・ハマりどころ表・アンチパターン・複数の例は、**それぞれが本当に価値を足す時だけ**入れる。下記の「よく使う要素」は道具箱であって、全部埋めるチェックリストではない。迷ったら削る。

### 粒度を対象で使い分ける

- **概念・仕組み・全体像・なぜ(why)**: 理解に必要なだけ書いてよい。図・段階説明・トレードオフが効く。
- **個別 API(メソッド / 演算子 / 関数 / フック / オプション)**: 「何で・何ができるか」が分かる最小限。1〜数行 + 必要なら小さい例 1 つ。1 つの API に何十行も使わない。

悪い例(`satisfies` 1 演算子に比較表 + 用途 1 + 用途 2 + ハマりどころ表 + 例 4 つ = 数十行)→ 過剰。
良い例:

```markdown
### `satisfies T`(TS 4.9)

値が型 T に代入可能かを**検査だけ**し、型は推論された具体型のまま(`as` と違い嘘は通らない / 注釈 `: T` と違い literal が消えない)。Record のキー網羅チェックと値の literal 維持を両立したい時に使う。

\`\`\`ts
const labels = { OWNER: 'オーナー' } satisfies Record<Role, string>;
labels.OWNER; // 型は 'オーナー'(literal 維持)。Role の漏れは satisfies がエラーに
\`\`\`
```

ユーザーが「○○との違いも」と明示した時だけ、その比較を 1 文か小さな表で最小限に添える。

## 文体

- **だ・である調 + 体言止め**の簡潔なメモ調。ですます調にしない。
- 1 文を短く。冗長な前置き・あいさつ・要約のための要約を書かない。
- 重要語は `**太字**`、コード/識別子/ファイル名は `` `バッククォート` ``。
- 因果・帰結は `→` で繋ぐ(例: 「readonly で受ける → 中で sort しようとすると TS エラー」)。
- 良し悪し・可否は `✅` / `❌` で示す(コード例のコメント末尾など)。
- 「初学者の自分が後で読み返して理解できる」密度。概念の **why**(なぜそうするか)を一言添える。

## ファイルとタイトル

- 1 ファイル = 1 トピック。ファイル名は `<トピック>.md`(例: `SEO.md`, `Stripe.md`)。
- 先頭行は `# <トピック> メモ`(例: `# SEO メモ`)。
- タイトル直後に**定義段落**(2〜4 文)。「それが何か」「どう使う/どこで効くか」「本ノートのスコープ」を書く。スコープを絞った場合は明記する(例: 「本ノートは実装側に絞る」)。

## 全体構成

```
# <トピック> メモ

<定義段落(2〜4 文)>

---

## 1. <全体像 or 基礎>

<本文・表・コード>

---

## 2. <次のテーマ>

...

---

## N. アンチパターンと注意点   ← 広いトピックで価値がある時だけ

<表: アンチパターン | リスク | 正しい対処>

---

## N+1. 参考リンク   ← 公式ドキュメントを残す価値がある時だけ

- <名前>: <公式 URL>
```

- セクションは `## N. 見出し` の**連番**。サブセクションは `### 見出し`(番号なし)。
- セクション間は `---`(水平線)で区切る。定義段落の直後にも `---` を置く。
- 「全体像 → 各論 → 実装パターン → アンチパターン → 参考リンク」は**広いトピックを 1 ノートに起こす時の最大構成**。小さな概念の追記では `## N. <概念>` 1 つ + 例 1 つで終えてよい。アンチパターン節・参考リンク節は義務ではない。

## よく使う要素(道具箱 — 必要な時だけ使う)

### 表

比較・対応関係・フィールド一覧で**情報が密に詰まる時**に Markdown table にする。1〜2 項目の差なら箇条書きや 1 文で済ませる方が端的。

```markdown
| 書き方 | 同じ意味 | 主な使い所 |
| --- | --- | --- |
| `readonly T[]` | 推奨(短い・現代的) | 関数パラメータ、型注釈 |
```

### コードブロック

必ず言語を指定する(`ts` / `tsx` / `html` / `bash` / `sql` 等)。コメントで補足し、`✅` / `❌` を行末に置いて可否を示す。

```ts
arr.push(4);   // ❌ Property 'push' does not exist
arr.map(x => x); // ✅ 新しい配列を返すメソッドは OK
```

### 図(ASCII)

データの流れなどはコードブロック内の簡易図で示す(Stripe ノートの「自前アプリ ──> Stripe」のような矢印図)。

### 用途・パターン提示

実用パターンを示す時は「### 用途 N: ...」の見出しで、危ない例(❌)→ 安全な例(✅)→ 一言の効能、の順にすると既存ノートと揃う。ただし**用途を何個も並べない** — 最も代表的なものを 1 つ示せば十分なことが多い。

## 追記する場合

- まず既存ノートを Read し、章立てと既存内容を把握する。
- 新トピックの章を足すなら、内容的に近い既存セクションの隣に置き、**連番を振り直す**(以降のセクション番号がズレる点に注意。番号参照が本文にあれば併せて直す)。
- 既存セクション内の小トピック追加なら `###` サブセクションとして差し込む。
- 既存の文体・粒度に合わせる。前後のセクションを読んでトーンを合わせること。

## やらないこと

- ですます調にしない。
- プロジェクト固有の進捗・仕様・リポジトリ名・社名・秘密情報を書かない(汎用知識に一般化する)。
- 既存ノートと重複する内容を足さない。
- 過剰な装飾・絵文字の乱用をしない(`✅` `❌` `→` 程度に留める)。
