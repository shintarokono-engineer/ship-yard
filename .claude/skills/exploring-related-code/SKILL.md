「機能の入り口」から関連ファイルを辿り、構造を散文 + 図で示します。網羅ではなくツアーです。

## いつ使うか

- 実装着手前に、影響範囲のコードを把握したい。
- リファクタ前に、対象領域の依存関係を可視化したい。
- 参画直後のオンボーディング、または「このコードベースはどう構成されているか?」と問われた時。
- 特定機能（auth / 決済 / データフロー など）を端から端まで理解したい時。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **対象機能/領域**: 機能名 / エントリポイントの URL や route / 着手予定のチケット
   - **入口の手がかり**: ファイルパス既知 / ルーティング定義から / 主要コンポーネント名
   - **出力形式**: 散文のみ / ディレクトリツリー / Mermaid 図 / 全部
   - **深さ**: トップ 2 階層 / 関数レベルまで / データフローを末端 DB まで追う
   - **特殊観点**: auth フロー / データフロー / テストピラミッド / 一般構造

2. **トップレベル構成を把握**:
   - `Glob` でリポジトリルートを列挙
   - `src/`, `app/`, `packages/`, `apps/`, `services/` などのエントリ候補を特定
   - `package.json` / `tsconfig.json` / `next.config.*` から建付けを判定

3. **入口から依存を辿る**:
   - エントリファイルを `Read` し、import を辿る
   - 最低 1 ファイルずつ実際に開いて責務を確認（捏造禁止）
   - 主要モジュール 5〜10 個に絞る

4. **構造を可視化**:
   - **ディレクトリツリー**: 主要フォルダ + 1 行責務
   - **Mermaid 図**: 意味のある矢印が 5 本以上あるなら描く。少なければ散文に留める
   - **規約メモ**: 命名・テスト位置・型置き場など気付いたこと

5. **成果物を `.claude/output/exploring-related-code/<YYYY-MM-DD-HHmm>-<topic>.md` に書き出し** — 標準出力にも要約 + どこから読み始めるかの推奨ステップ。

## 出力フォーマット

```markdown
# 探索対象: <機能/領域名>

## アーキテクチャ

<1 段落: SPA / モノレポ / RSC など、主要 FW、パッケージマネージャ>

## 構成

\`\`\`
project/
├── src/
│ ├── components/ # 表示用 UI 部品
│ ├── features/ # 機能スコープのロジック
│ ├── hooks/ # 共有フック
│ └── api/ # API クライアント + 型
└── tests/
\`\`\`

## モジュール依存（必要に応じて）

\`\`\`mermaid
flowchart LR
components --> hooks
features --> components
features --> api
\`\`\`

## 知っておくべき規約

- <事項>: <場所>

## どこから読み始めるか

1. `<エントリポイント>` — 機能ブート
2. `<主要 feature フォルダ>` — 1 機能を端から端まで
3. `<テストセットアップ>` — テスト方針
```

## ルール

- **ファイル全部を列挙しない**。中身が予測可能なディレクトリで止める。
- **読んで検証する**。「API クライアント」と呼ぶなら最低 1 ファイル開いて確認。
- **不明点は不明と述べる**。「`src/foo/` の目的を判定できなかった — 3 ファイルで関心事が混在」。捏造より良い。
- **Mermaid は明確化に寄与するときだけ**。意味ある矢印が 5 本未満なら散文で。
- マップ全体を 1 画面以内に収める。巨大なら 1 スライスずつ提示し、どのスライスを見たいかユーザーに尋ねる。

## バリエーション

- **「auth フローをマップして」** — login → session → 保護ルート。シーケンス図でレンダー。
- **「データフローをマップして」** — DB → API → クライアント。永続層を下に置いた flowchart。
- **「テストピラミッドをマップして」** — unit / integration / e2e フォルダと件数。

各バリエーションでも同じ流れ: 正確になる程度に読み、最小限にレンダー、正直にラベル付け。
