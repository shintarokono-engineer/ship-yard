エラーから「症状 → 仮説 → 検証 → 根本原因 → 修正方針」の順に組み立てます。修正そのものはこの skill のスコープ外。

## いつ使うか

- バグ報告を受け取った直後。
- 本番 / staging で例外を観測した時。
- 開発中に未知のエラーで詰まった時。
- **使わない**: テスト失敗の調査専用（→ `debugging-failing-test`）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける。情報不足は仮説の質を直撃するので妥協しない:
   - **エラー本文**: 完全なスタックトレース / メッセージのみ / ログ抜粋
   - **再現性**: 100% / 散発的 / 一度きり
   - **再現手順**: 既知 / 不明 / 推測
   - **環境**: ローカル / staging / production / CI
   - **発生条件**: 最近の変更後 / 特定ユーザー / 特定データ / 時間帯依存
   - **影響範囲**: 単機能 / 全ユーザー / 一部ユーザー / データ破損リスクあり
   - **緊急度**: 即対応 / 営業日対応 / 調査のみ
   - **既に試したこと**: なし / 一部仮説検証済み

2. **症状を構造化**:
   - エラー種別（型エラー / null参照 / 認可 / ネットワーク / etc.）
   - 発生箇所（ファイル:行 / モジュール）
   - スタックトレースから直前の呼び出し経路
   - 関連ログの時系列

3. **仮説を 2〜4 個立てる**:
   - 各仮説に「もしこれなら、X が観察されるはず」の検証ポイントを添える
   - 仮説の優先度（尤度 + 検証コスト）

4. **検証手順を提示**:
   - 各仮説を最小コストで切り分ける手順
   - ログ追加 / 再現スクリプト / git blame / 直近変更の確認 など
   - 必要ならコード調査（`exploring-related-code` を呼ぶ）

5. **根本原因が判定できたら修正方針**:
   - 最小修正 / 構造的修正の選択肢
   - regression テストの観点（修正と同時に追加すべきテスト）

6. **成果物を `.claude/output/investigating-error/<YYYY-MM-DD-HHmm>-<error-summary>.md` に書き出し** — 標準出力には仮説サマリと推奨次手順のみ。

## 出力フォーマット

```markdown
# 調査: <エラー1行サマリ>

## 症状

- **エラー**: `<例外名 / メッセージ>`
- **発生箇所**: `<file:line>` (関数名)
- **再現性**: <100% / 散発 / 一度きり>
- **環境**: <production / staging / local>
- **影響**: <ユーザー数 / 機能範囲>

## スタックトレース要点

\`\`\`
<関連箇所のみ。ノイズは省略>
\`\`\`

## 仮説

### 仮説 1: <短いタイトル>（尤度: 高/中/低）

**根拠**: <なぜそう思うか>
**検証**: <最小コストで真偽を分ける手順>
**もし真なら**: <根本原因 + 修正方針>

### 仮説 2: ...

## 切り分け推奨順序

1. 仮説 1 を <手順> で確認
2. 否定されたら仮説 2 へ

## 既知の代替策（恒久対応までの一時しのぎ）

- <あれば>

## 次のステップ

- <仮説検証 / コード調査 / 関係者へエスカレーション>
```

## ルール

- **症状と仮説を分離する**。スタックトレースの記述を「原因」と書かない。
- **仮説は複数立てる**。最初の 1 つに飛びつかない。
- **検証コストを意識**。grep 1 回で潰せる仮説から先に。
- **修正コードを書かない**。調査結果まで。修正は別の skill / agent / 手動で。
- **不明な情報は不明と書く**。捏造より価値あり。
- regression テストの観点は必ず提示する（修正と同時に追加されないと再発する）。
