開発ループの最初のステップ。曖昧なチケットを「実装可能な要件」に翻訳します。

## いつ使うか

- 新しい issue / チケット / Slack 依頼を受け取った直後。
- 既存チケットの再点検（仕様確定前の最終確認）。
- **使わない**: 既に要件が完全に固まっていて疑問なし、かつ単純なバグ修正の場合。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **チケット**: URL / ID / 本文貼り付け / 口頭の依頼内容
   - **既知の制約**: スケジュール / 依存タスク / 関連 PR
   - **不明点**: ユーザー側で既に疑問があるか
   - **受け入れ条件の認識**: 明示的にチケットにあるか / 自分で導出する必要があるか
   - **想定読者**: 自分用メモ / チームレビュー用 / 上長承認用

2. **チケット読解**:
   - URL がある場合は MCP ツール（GitHub / Linear / Jira など）で fetch
   - 本文貼り付けの場合はそのまま読む
   - 関連 issue / PR / ADR / コメント履歴も追えるなら追う

3. **要件の構造化**:
   - 「やりたいこと」「やらなくてよいこと」「やってはいけないこと」を分離
   - 受け入れ条件を 3〜7 個に分解（チェックリスト形式）
   - 影響範囲の初期見立て（フロントのみ / バックのみ / 両方 / インフラ含む）

4. **不明点を列挙** — 着手前に潰すべき疑問を並べる:
   - 仕様の曖昧さ
   - エッジケース未定義
   - 依存先の実装状況
   - 性能/セキュリティ要件の明示有無

5. **成果物を `.claude/output/understanding-ticket/<YYYY-MM-DD-HHmm>-<ticket-id>.md` に書き出し** — 標準出力にも要約。

## 出力フォーマット

```markdown
# <ticket-id>: <タイトル>

## サマリ

<1〜2 行で「何を作るか」>

## やること / やらないこと

**やること**:

- ...
  **やらないこと**:
- ...

## 受け入れ条件（解釈版）

- [ ] ...
- [ ] ...

## 影響範囲（初期見立て）

- フロント: <ファイル/領域>
- バック: <ファイル/領域>
- DB / インフラ: <あれば>

## 不明点（着手前に潰すべき）

1. ...
2. ...

## 次のステップ

- 不明点 1, 2 を担当者/PdM に確認
- 確認取れ次第 `planning-implementation` skill で実装計画を立てる
```

## ルール

- **チケット本文を捏造しない**。読めなかった場合は「読めなかった」と書く。
- **解釈と引用を分ける**。受け入れ条件の解釈版とチケット原文の差分は明示。
- **「やらないこと」を必ず書く**。スコープ拡大の予防。
- 不明点は質問形式で、答えやすく書く（YES/NO で答えられる粒度を意識）。
