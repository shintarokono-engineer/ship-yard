「これ入れて大丈夫?」を、感覚ではなく材料で判断するための skill です。

## いつ使うか

- 新しい依存パッケージの追加検討時。
- 既存パッケージのメジャーアップデート前。
- 既存パッケージを別のものに置き換える検討時。
- **使わない**: バンドル分析の詳細解析（→ 別ツール）/ 既知脆弱性の自動修正（→ `npm audit fix` 等）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **候補パッケージ**: 名前 / npm/pypi の URL
   - **用途**: 何の問題を解決したいか
   - **代替検討状況**: 標準 API で代替可能か検討済み / 候補が複数ある / 既に決め打ち
   - **環境**: dev 依存（test/build） / 本番依存
   - **リスク許容度**: 厳格（OSS 監査必須） / 標準 / 緩い（PoC など）
   - **既知の制約**: ライセンス（GPL 不可など） / バンドルサイズ上限 / Node/Python バージョン

2. **情報収集**:
   - npm/pypi ページから: 最新版・更新頻度・週間 DL 数・メンテナ
   - GitHub から: スター数 / open issue 数 / 最終コミット / リリース頻度
   - ライセンス（package.json / pyproject.toml / LICENSE ファイル）
   - 依存ツリー: `npm view <pkg> dependencies` / `pipdeptree`
   - サイズ: `bundlephobia.com` 相当の指標 / `pip show <pkg>`
   - 既知脆弱性: `npm audit` / `pip-audit` / GitHub Advisory

3. **代替候補を 1〜2 個挙げる**:
   - 標準 API / 既存依存で代替可能か
   - 同等機能の別パッケージ
   - 自前実装の現実性（線数、保守コスト）

4. **総合判定**:
   - **入れてよい** — 理由付きで
   - **要注意** — 条件付き
   - **入れない方がよい** — 代替案を提示

5. **成果物を `.claude/output/reviewing-dependency/<package-name>.md` に書き出し** — 標準出力には判定とその根拠 3 行のみ。

## 出力フォーマット

```markdown
# 依存レビュー: <package-name>@<version>

## 用途
<このパッケージで解決したいこと>

## 基本情報
- **最新版**: <version> (<date>)
- **ライセンス**: <MIT / Apache-2.0 / etc.>
- **週間 DL**: <number>
- **GitHub**: <stars> stars, <open-issues> open issues, last commit <date>
- **メンテナ**: <個人 / 企業 / コミュニティ>
- **依存数**: 直接 <N>, 推移的 <M>

## サイズ
- **インストール後**: <KB / MB>
- **gzipped**: <KB>（フロント依存の場合）
- **tree-shakable**: <Yes/No>

## 既知脆弱性
- <CVE 一覧 or なし>

## ライセンス整合性
- 本プロジェクトとの相性: <OK / 要確認>
- 制約事項: <copyleft 等>

## 代替候補

| 候補 | 利点 | 欠点 |
|---|---|---|
| 標準 API (`<API>`) | 依存なし | <制約> |
| `<alt-package>` | <利点> | <欠点> |
| 自前実装 | 完全制御 | <line 数 / 保守コスト> |

## 判定

**結論**: <入れてよい / 要注意 / 入れない方がよい>

**根拠**:
1. ...
2. ...

## もし入れる場合の手順
1. <install コマンド>
2. <設定追加 / 型定義 / etc.>
3. <peer dependency 確認>
```

## ルール

- **メンテ停止疑いを軽視しない**。最終コミットが 1 年以上前 / open issue が放置 → 要注意。
- **ライセンス確認は省略しない**。GPL 系を本番依存に入れる前は法務 / リード相談。
- **代替候補を必ず 1 つ以上提示**。「これしかない」は判断材料を狭める。
- **判定は明示**（「入れてよい / 要注意 / 入れない方がよい」のいずれか）。曖昧な「ケースバイケース」は避ける。
- **入れない判定でも理由を必ず書く**。後で再検討する人のため。
- 値（DL 数 / star 数 / CVE）は確認した値を書く。捏造禁止。
