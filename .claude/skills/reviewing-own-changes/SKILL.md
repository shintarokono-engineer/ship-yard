差分または指定ファイル群を **全文** 読み、★ 評価付きの構造化レビューを返します。コード修正は別。

## いつ使うか

- PR を出す直前のセルフレビュー。
- 特定ファイル群の品質チェック（diff 外の既存コードも対象）。
- 設計思想ドキュメントとの突合せレビュー。
- **使わない**: コミット前のテスト/lint チェック（→ `checking-commit`） / 他人の PR の汎用レビュー（→ `/review` ビルトイン or `code-reviewer` agent）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **対象モード**: 差分（local / PR）/ 指定ファイル群 / 両方
   - **差分対象**: 現在のブランチ vs `main` / PR 番号（`gh pr diff <num>`）/ 指定ブランチ
   - **指定ファイル**: パス一覧 / Glob パターン
   - **設計思想ドキュメント**: パス指定（例: `docs/web.instructions.md`, `docs/service.instructions.md`） / なし（汎用観点でレビュー）
   - **観点重み**: 全部均等 / 性能 / セキュリティ / 可読性 / アクセシビリティ
   - **追加観点**: TypeScript 厳格性 / 後方互換 / Figma 連携（UI なら URL ヒアリング）/ プロジェクト固有
   - **事前情報**: 関連チケット ID / 想定レビュアー / 急ぎ度

2. **対象を取得**:
   - 差分モード:
     ```bash
     git diff <base>...HEAD --stat       # 概要
     gh pr view <num>                     # PR メタ情報
     gh pr diff <num>                      # PR 差分
     ```
   - **重要**: 差分モードでも、touch されたファイルは **全文を `Read`** する（差分のスニペットだけで判断しない）
   - 指定ファイルモード: 指定ファイル/Glob を全文 `Read`

3. **設計思想ドキュメントを読む**（指定があれば）:
   - 例: `web.instructions.md`, `basic-design.instructions.md`, `service.instructions.md`, `bff.instructions.md`
   - 評価基準として後段の準拠チェックで参照
   - **指定なしなら汎用観点でレビュー** し、その旨を出力に明示

4. **観点別にチェック**（重み設定に応じて取捨）:
   - **命名 / 設計**: 単一責任、不要な抽象、命名の意図、`data` / `info` / `manager` の濫用
   - **可読性 / 保守性**: 入れ子の深さ、関数長、責務の混在
   - **テスト**: 網羅性、エッジケース、テストが「実装」でなく「振る舞い」を見ているか
   - **性能**: N+1、不要な再レンダー、O(n²) の潜在
   - **セキュリティ**: 入力検証、認可チェック、シークレット混入、SQL/XSS
   - **後方互換**: 公開 API / DB スキーマ / 設定形式
   - **エラーハンドリング**: 握りつぶし、型情報の欠落、ユーザー向けメッセージ
   - **TypeScript**: `any` 排除、型表現の的確性、判別共用体の活用
   - **アクセシビリティ**（UI 変更時）: ラベル、キーボード、フォーカス、セマンティック HTML
   - **設計思想準拠**: 該当ドキュメントの指針との整合性

5. **ファイルごとに ★ 評価**: 全文を読んだ上で 5 段階評価 + 良い点 + 改善点（コード例付き）。

6. **優先度別に集約**: 🔴 高 / 🟡 中 / 🟢 低。

7. **設計思想準拠状況**: ✅ 良好 / ⚠️ 要注意 を分けて列挙。

8. **成果物を `.claude/output/reviewing-own-changes/<YYYY-MM-DD-HHmm>-<branch-or-topic>.md` に書き出し** — 標準出力にも完成版を提示。

```markdown

<対象全体の品質サマリ。設計思想への準拠度を含めて 2〜3 文>

## ファイル別詳細レビュー

### `path/to/file.ts` — ⭐⭐⭐⭐☆ (良好)

**良い点:**

- <具体的な良い実装>
- <設計思想に沿った部分>

**改善点:**

- **`path/to/file.ts:42`** — <問題> → <修正案>
- ...

\`\`\`typescript
// 現在の実装（問題のある例）
const example = '問題のあるコード';

// 推奨: 改善後の実装
const example = '改善されたコード';
\`\`\`

---

### `path/to/other.tsx` — ⭐⭐⭐☆☆ (要改善)

**良い点:**

- ...

**改善点:**

- ...

\`\`\`tsx
// 現在
<button onClick={handler}>Submit</button>

// 推奨: aria-label とキーボード操作の確保
<button onClick={handler} aria-label="送信">Submit</button>
\`\`\`

---

## 優先度別改善提案

### 🔴 高優先度

1. **`path/to/file.ts:42`** — <要約> → <修正案>

### 🟡 中優先度

1. **`path/to/file.ts:108`** — <要約> → <修正案>

### 🟢 低優先度

## 設計思想への準拠状況

> 参照ドキュメント: `<指定パス>` / なし

✅ **良好な点:**

- <設計思想に沿った実装>

⚠️ **要注意点:**

- <設計思想から逸脱している部分>
```

## ★ 評価基準

| 評価               | 意味                             |
| ------------------ | -------------------------------- |
| ⭐⭐⭐⭐⭐ (優秀)  | 設計思想に完全準拠、問題なし     |
| ⭐⭐⭐⭐☆ (良好)   | 軽微な改善余地あり               |
| ⭐⭐⭐☆☆ (要改善)  | 明確な問題あり、修正推奨         |
| ⭐⭐☆☆☆ (問題あり) | 重大な問題、即座に修正必要       |
| ⭐☆☆☆☆ (不適切)    | 設計思想に反する、全面見直し必要 |

## ルール

- **指摘は `file:line` で引用**。「全体的に〜」は避ける。
- **改善提案にはコード例を必ず添える**（高/中優先度）。`// 現在 → // 推奨` の対比形式が望ましい。
- **★ 評価は対象ファイル全文を読んだ上で付ける**。差分のスニペットだけ見て付けない。
- **設計思想ドキュメントが指定されたら必ず参照**。なければ「設計思想ドキュメント未参照（汎用観点で評価）」と出力に明示。
- **Figma 連携が必要な UI レビューでは URL をユーザーに求める**（MCP 経由で参照可能なら）。
- **水増ししない**。問題がなければ「問題なし」と書いて終わる。
- **修正コードを直接適用しない**。提案のみ。実装は別途。
- **自分が書いたコードへの遠慮を排除**。他人の差分のつもりで読む。
- ★ 評価は厳格に付ける。⭐⭐⭐⭐⭐ は本当に問題なしの時だけ。デフォルト ⭐⭐⭐⭐☆ で始める癖をつける。
