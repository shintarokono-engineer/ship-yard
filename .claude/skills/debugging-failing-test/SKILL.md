失敗テストの「症状 → 切り分け → 修正方針」を体系化します。

## いつ使うか

- ローカル / CI でテストが落ちた時。
- 「先週まで通っていたのに突然落ち始めた」テストの調査。
- フレーキーテスト疑いの確認。
- **使わない**: 一般的なエラー調査（→ `investigating-error`） / テスト新規作成（→ `test-writer` agent）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **失敗テスト**: ファイル:テスト名 / コマンド出力貼り付け / CI のログ URL
   - **再現性**: ローカルで毎回失敗 / CI のみ失敗 / 散発的（フレーキー疑い）
   - **直近の変更**: 該当ファイル周辺の最近の差分 / 全く触っていない
   - **環境差分**: ローカル/CI のバージョン違い / DB / 時刻 / タイムゾーン
   - **テストフレームワーク**: Jest / Vitest / Playwright / pytest / etc.
   - **緊急度**: マージブロック中 / 後回し可

2. **失敗の症状を取得**:
   - スタックトレース全文
   - 期待値 vs 実際値（assertion error の場合）
   - 失敗テストの周辺（before/afterEach、fixture）を `Read`
   - 直近のコミット履歴: `git log <test-file>` で「いつから壊れているか」

3. **切り分け（4 分類）**:
   - **テストが間違っている**: 仕様変更に追従できていない / 古い前提
   - **実装が間違っている**: テストの期待が正しい、実装にバグ
   - **環境問題**: DB 状態、シーディング、外部依存、時刻、TZ
   - **フレーキー**: 並行性、順序依存、async 待ち合わせ、リソース競合

4. **検証手順**:
   - 単独実行で再現するか: `npx vitest run <file>`
   - 順序依存疑いなら順番を変えて実行
   - 環境疑いなら `--shard`、`--seed`、CI 同等イメージで実行
   - フレーキー疑いなら 10 回ループ

5. **修正方針**:
   - 4 分類のどれかを判定
   - テスト修正案 / 実装修正案 / 環境修正案 / 安定化（retry / 待ち合わせ修正）
   - characterization テスト追加が必要なら言及

6. **成果物を `.claude/output/debugging-failing-test/<YYYY-MM-DD-HHmm>-<test>.md` に書き出し** — 標準出力には判定 + 推奨次手順のみ。

## 出力フォーマット

```markdown
# 失敗テスト調査: <test-name>

## 症状
- **テスト**: `<file:test>`
- **失敗内容**: `<assertion error / timeout / setup error>`
- **再現性**: <ローカル毎回 / CI のみ / 散発>
- **直近変更**: <あれば commit/PR>

## 失敗詳細
\`\`\`
<関連箇所のみ>
\`\`\`

## 判定

**カテゴリ**: <テストが間違い / 実装が間違い / 環境 / フレーキー>

**根拠**:
- ...

## 修正方針

### 推奨修正
- **対象**: <test ファイル / 実装ファイル / CI 設定>
- **変更**: <具体的修正案>

### 代替案
- ...

## 切り分けが追加で必要なら
1. <検証コマンド>
2. ...

## 次のステップ
- <修正実装 / さらなる切り分け / マージブロック解除のための一時 skip>
```

## ルール

- **テストを安易に skip しない**。skip するなら期限と issue 番号を必須に。
- **フレーキー判定は 10 回以上の試行が必要**。1 回成功で「直った」と言わない。
- **assertion を緩めて通さない**。実装が正しいか先に確認。
- **「動いたから OK」を避ける**。なぜ動くようになったかを必ず言語化。
- 修正コードを直接書かない。修正方針まで。実装は別の skill / agent / 手動で。
