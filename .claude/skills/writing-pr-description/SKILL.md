レビュアーが 30 秒で全体像を掴めて、3 分で詳細を追える PR 本文を作ります。

## いつ使うか

- 新規 PR を出す直前。
- ドラフト PR の本文を ready 用に書き直すとき。
- 既存 PR 本文を改善したいとき。
- **使わない**: コミットメッセージ生成（PR 本文とは別）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **対象**: 現在のブランチ / 指定 PR 番号（`gh pr view`） / branch 名直指定
   - **比較先**: `main` / `develop` / その他
   - **トーン**: 簡潔（数行 + 箇条書き） / 詳細（背景含む） / 既存テンプレ準拠
   - **必須セクション**: What/Why/How / テスト方法 / スクリーンショット / リスク&ロールバック / 関連 issue
   - **読者層**: チーム内（前提共有あり） / 横断レビュー（前提なし）
   - **関連リンク**: チケット ID / Figma / ADR / 他 PR

2. **差分とコミット履歴を取得**:

   ```bash
   git diff <base>...HEAD --stat
   git log <base>..HEAD --oneline
   gh pr view <num>            # 既存 PR の場合
   gh pr diff <num>             # 同上
   ```

3. **What / Why / How を整理**:
   - **What**: この PR で何が変わったか（読者が真っ先に知りたい）
   - **Why**: なぜこの変更が必要か（チケット背景の要約）
   - **How**: 実装方針の要点（詳細はコードに任せる）

4. **テスト方法と検証手順** — レビュアーが追試できる粒度で:
   - 自動テストでカバーされる範囲
   - 手動検証手順（必要なら）
   - スクリーンショット / 動画（UI 変更の場合）

5. **リスクとロールバック**:
   - 想定する破壊的変更
   - rollback 手順（reverse の git revert で済むか / DB マイグレーションも含むか）

6. **成果物を `.claude/output/writing-pr-description/<YYYY-MM-DD-HHmm>-<branch>.md` に書き出し** — 標準出力にも完成版を提示。

## 出力フォーマット

```markdown
## Summary

<1〜3 行: 何をしたか>

## Why

<1〜3 行: なぜやったか。チケット要約 or 背景>

Closes #<issue-id>

## How

- <主要な変更 1>
- <主要な変更 2>
- <設計判断のポイント>

## Test plan

- [ ] <自動テストの観点>
- [ ] <手動検証手順>
- [ ] <UI 変更がある場合: 主要画面のスクショ>

## Screenshots / Demo

<UI 変更がある場合は画像/動画>

## Risks & Rollback

- **リスク**: <影響範囲>
- **Rollback**: <git revert / マイグレーション戻し手順>

## Related

- Ticket: <ID/URL>
- Design: <Figma URL>
- ADR: <doc 参照>
```

## ルール

- **コミット履歴をそのまま貼らない**。要約する。
- **コードコメントを再生産しない**。なぜを書き、何はコードに任せる。
- **スクショ枠を空のまま残さない**。UI 変更がなければセクションごと削除。
- **想定読者に合わせる**。横断レビューなら背景を厚めに、チーム内なら簡潔に。
- 関連リンクは検証して書く。捏造しない（issue 番号など特に）。
