レビューコメントを「対応する/しない」と「どう答えるか」に分解します。実装は別。

## いつ使うか

- PR レビューを受けてコメントが付いた直後。
- 大量コメントが付いて優先順位を整理したい時。
- **使わない**: 自分の差分のセルフレビュー（→ `reviewing-own-changes`） / 単一の trivial コメント対応（即修正してよい）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **対象 PR**: PR 番号 / URL / `gh pr view <num> --comments` で取得
   - **コメント**: 全件 / 未対応のみ / 特定コメント ID
   - **同意/反論の方針**: 全部受け入れる / 議論したい点あり / レビュアー意図不明な箇所あり
   - **回答スタイル**: 簡潔（OK / 修正済み） / 丁寧（背景含む） / 反論時は根拠を厚めに
   - **緊急度**: 当日中 / 数日OK

2. **コメント取得**:

   ```bash
   gh pr view <num> --comments
   gh api repos/<owner>/<repo>/pulls/<num>/comments
   ```

3. **コメントを分類**:
   - `[修正]` 同意して直す
   - `[反論]` 同意しない、議論で押し戻す
   - `[質問]` 意図確認が必要
   - `[別PR]` 同意するが本 PR スコープ外、別 PR で
   - `[既対応]` 既に直した / 既に対応済みであることを伝えるだけ

4. **各コメントに返信案を作成**:
   - 修正系は「修正済み (commit `<sha>`)」or「修正します」
   - 反論系は根拠 + 代替案を提示
   - 質問系は YES/NO で答えやすい形に整理

5. **修正必要なコメントの実装計画**:
   - ファイル単位でグルーピング
   - 影響が大きいなら `planning-implementation` に渡す候補

6. **成果物を `.claude/output/responding-to-review/<YYYY-MM-DD-HHmm>-pr<num>.md` に書き出し** — 標準出力には分類サマリと「次に直すべきファイル」のみ。

## 出力フォーマット

```markdown
# PR #<num> レビュー対応

## サマリ

- コメント数: <N> ([修正] X, [反論] Y, [質問] Z, [別PR] W, [既対応] V)

## [修正]

### <file:line> @reviewer

> <元コメント要約>

**対応**: <修正方針>
**返信案**:

> 修正します。<追加コメントあれば>

### ...

## [反論]

### <file:line> @reviewer

> <元コメント要約>

**理由**: <なぜ反論するか、根拠>
**返信案**:

> <丁寧な反論文。代替案を含む>

## [質問]

### <file:line> @reviewer

> <元コメント要約>

**確認したいこと**: <質問の要点>
**返信案**:

> <YES/NO で答えやすい質問形式>

## [別PR]

### <file:line> @reviewer

> <元コメント要約>

**返信案**:

> 同意します。スコープ外なので別 PR (<issue-id>) で対応します。

## [既対応]

### <file:line>

**返信案**: 修正済みです（commit <sha>）

## 修正実装の優先順位

1. <file>: <修正内容> — blocker
2. <file>: <修正内容> — nit
```

## ルール

- **「修正します」と書いたら必ず修正する**（または別 PR を作る）。約束を放置しない。
- **反論は丁寧に**。レビュアーの意図を尊重した上で根拠を述べる。
- **質問は YES/NO 形式に整理**。レビュアーの返信コストを下げる。
- **同意できないコメントを黙殺しない**。明示的に反論するか別 PR にする。
- 修正コミットは「<コメント参照> に対応」など短い説明を付ける。
