「何を確認すれば動作担保できるか」を観点別に列挙し、checkbox 形式の markdown で返します。コピペでそのまま使える形式が必須。

## いつ使うか

- 実装完了後、PR の Test plan セクションを埋める時。
- QA / 別エンジニアへの動作確認引き継ぎ時。
- 自分でセルフ動作確認する前のチェックリスト作成時。
- リリース前の最終確認観点を整理する時。
- **使わない**: 自動テストの設計（→ `test-writer` agent）/ PR 説明書き全体の生成（→ `writing-pr-description`）。

## 進め方

1. **必要情報をヒアリング** — `AskUserQuestion` で順次。最初の指示に含まれている項目はスキップ。納得まで掘り下げ可。1 ラウンドあたり最大 4 質問のため、項目が多い場合は複数ラウンドに分ける:
   - **対象**: 機能名 / 対象差分（branch / PR 番号） / チケット ID
   - **観点重み**: golden path のみ / エッジケース重視 / 性能含む / セキュリティ含む / a11y 含む / 全部
   - **環境**: ローカル / staging / production / 全環境
   - **想定実行者**: 自分（開発者） / QA / 別エンジニア / 横断レビュー
   - **既存テストとの棲み分け**: 自動テストでカバー済の観点は除外 / カバー済も含めて全列挙
   - **特殊観点**: モバイル / マルチブラウザ / ダークモード / 多言語 / 大量データ

2. **対象の差分/機能を読み込む**:
   - 差分ベースなら: `git diff <base>...HEAD` / `gh pr diff <num>` / `gh pr view <num>`
   - 機能ベースなら: 該当ディレクトリを `Glob` / `Grep` で確認
   - UI 変更を判定: コンポーネント / ルーティング差分の有無

3. **観点をカテゴリ別に列挙**:
   - **Golden path** — 想定通りの正常系
   - **エッジケース** — 空・上限・不正・タイムアウト・並行性
   - **エラーハンドリング** — エラー表示・リトライ・フォールバック
   - **性能** — 応答時間 / 大量データ時の挙動 / N+1
   - **セキュリティ** — 認可境界 / 入力検証 / XSS/CSRF
   - **アクセシビリティ** — キーボード操作 / スクリーンリーダー / フォーカス
   - **後方互換** — 既存ユーザーの体験が壊れていないか
   - **特殊観点** — モバイル / マルチブラウザ / ダークモード / 多言語

4. **observable な確認手順に落とし込む**:
   - 各観点を「何を実行 → 何を観察 → 何が成功条件か」が分かる粒度に
   - チェックボックスで O/X が付けられる粒度
   - 専門用語を避け、QA/別エンジニアにそのまま渡せる表現

5. **成果物を `.claude/output/writing-verification-checklist/<YYYY-MM-DD-HHmm>-<topic>.md` に書き出し** — 標準出力にも完成版 markdown をそのまま提示（ユーザーがコピペできる形式）。

## 出力フォーマット

**完成版の markdown（コピペ用）**:

```markdown
## 動作確認観点: <機能/差分名>

### Golden path
- [ ] <ユーザー操作> → <期待される結果>
- [ ] ...

### エッジケース
- [ ] 空入力 / 必須項目未入力 → <期待されるエラー表示>
- [ ] 上限値（<具体値>）→ <期待される挙動>
- [ ] 不正フォーマット → <期待されるエラー>
- [ ] タイムアウト / ネットワーク切断 → <期待される fallback>

### エラーハンドリング
- [ ] サーバーエラー (5xx) 時に <期待される表示 / リトライ>
- [ ] 認証切れ時に <期待される再ログイン誘導>

### 性能
- [ ] 100 件以上のデータで描画が破綻しない
- [ ] 主要操作のレスポンスが <閾値> 以内

### セキュリティ
- [ ] 未ログインユーザーが保護リソースにアクセスできない
- [ ] 他ユーザーのリソースを ID 直接指定で取得できない

### アクセシビリティ（UI 変更がある場合）
- [ ] キーボードのみで全操作が完了できる
- [ ] フォーカスリングが見える
- [ ] スクリーンリーダーで主要コンテンツが読み上げられる

### 後方互換
- [ ] 既存ユーザーの保存データが正常に表示される
- [ ] 既存 API の呼び出し方法が変わっていない

### マルチ環境（必要に応じて）
- [ ] Chrome / Safari / Firefox で同等動作
- [ ] iOS / Android で同等動作
- [ ] ダークモードで視認性 OK

---
**実行環境**: <local / staging / production>
**実行者**: <自分 / QA / 別エンジニア>
**関連 PR / チケット**: <ID/URL>
```

## ルール

- **コピペでそのまま使える形式を最優先**。装飾コメントを混ぜない。
- **observable な粒度に**。「正しく動くこと」のような抽象表現は禁止。「<操作> → <期待結果>」の構造を守る。
- **観点重みに応じてセクションを取捨**。a11y を含めない指定なら a11y セクション削除。
- **専門用語を避ける**（QA に渡す可能性を考慮）。CSRF / IDOR などは「他ユーザーの ID で〜」のように噛み砕く。
- **既存自動テスト被りを避ける**指定があれば、`Grep` で該当 test を確認してから観点列挙。
- **環境固有の値を埋める**: 「100 件以上」「5xx エラー」のように具体数を入れる。プロジェクトの SLA に合わせる。
- 観点を捏造しない。差分から読み取れない観点は「対象外と思われるが念のため」と明示する。
