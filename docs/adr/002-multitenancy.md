# ADR-002: マルチテナント分離方式

## ステータス

承認済み(2026-05-01)

## 背景・問題

Shipyard は個人開発者(=ユーザー)が複数のワークスペース(=テナント)を持ち、各ワークスペースに複数メンバーが所属する B2B SaaS 構造を持つ。テナント間のデータ完全分離が前提だが、運用効率と開発速度を両立する分離方式を選定する必要がある。

関連: ADR-001(技術スタック)

## 検討した選択肢

### A. Pool model(全テナント共有 DB、tenantId カラム識別)

- 1つの DB、1つのスキーマ、各テーブルに tenantId カラム
- テナント識別はアプリケーションロジック
- 例: Slack、Notion(初期)

### B. Bridge model(共有 DB、テナント毎スキーマ分離)

- 1つの DB、テナント毎に PostgreSQL スキーマを切る
- データ分離が物理的
- 例: 一部のエンタープライズ SaaS

### C. Silo model(テナント毎 DB)

- 完全独立 DB、最強の分離
- 例: エンタープライズ向け、規制業界

## 決定

**Pool model** を採用する。

## 理由

- 個人開発者向けで小規模テナントが多数発生する想定 → Silo/Bridge は運用コスト過大
- Pool model は Prisma Client Extension で tenantId 自動注入することで人為ミスを構造的に防げる
- マイグレーション・バックアップ運用が一元化、3週間 MVP に適合
- スケール時に問題が出れば一部テナントを Silo に切り出す Hybrid model に進化可能

棄却理由:

- **Bridge model**: Prisma が動的スキーマ切替に弱い、マイグレーション運用が重い
- **Silo model**: 個人開発者向けにコスト過大、マルチワークスペースのクロスクエリが困難

## 結果

### 良い影響

- 単一 DB で運用コスト最小化
- Prisma Client Extension でテナント漏洩を構造的に防止
- 既存ノウハウ(SKY 案件のマルチテナント設計経験)が活かせる
- 副業面談で「Pool model + Prisma Client Extension」のセットで設計判断を語れる

### 悪い影響・リスク

- Pool model はアプリケーションのバグでテナント漏洩しうる
  - 対策: コードレビュー必須、Raw SQL 禁止 lint ルール導入
- 大規模テナントが現れた場合の Noisy Neighbor 問題
  - 対策: 監視で検知、必要時 Silo 切り出し
- バックアップからの特定テナント復旧が困難
  - 対策: 定期論理バックアップ + テナント単位のエクスポート機能を将来追加

### フォローアップ

- Prisma Client Extension の実装と単体テスト整備(Day 5)
- Raw SQL 利用箇所を ESLint カスタムルールで検出
- テナント単位のリソース使用量モニタリング(将来)
- データ消去依頼(GDPR 対応)時の手順を文書化
- ScheduledJob 実行時のテナントコンテキスト確立フローを設計
