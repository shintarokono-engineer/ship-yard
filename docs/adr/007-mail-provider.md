# ADR-007: メール送信基盤の選定

## ステータス

承認済み(2026-05-12)

## 背景・問題

Shipyard はトランザクショナルメール送信が必要であり、最優先の用途はメンバー招待(`screen-flow.md` フロー3、`InvitationToken` モデル)。将来用途として課金通知(Stripe イベント連動)、パスワードリセット補助、その他アカウント系通知がある。

`screen-flow.md` フロー3の図には「SES 経由」と記述されているが、プロバイダの選定は未確定のままだった(`PROJECT_STATUS.md` §9.2)。

関連: ADR-001(技術スタック)、ADR-004(課金プラン)

## 検討した選択肢

| プロバイダ | 料金                                  | 無料枠                 | 特徴                                                          |
| ---------- | ------------------------------------- | ---------------------- | ------------------------------------------------------------- |
| Resend     | $0(無料枠内)/ $20 月〜(有料プラン)    | 100 通/日、3,000 通/月 | React Email 統合、API がシンプル、個人開発向け、即日利用可能  |
| Postmark   | $15/月〜(10,000 通)                   | 100 通(テスト用)       | 配送性能トップクラス、トランザクショナル特化、DKIM 設定が自動 |
| AWS SES    | $0.10 / 1,000 通(+ EC2 外へ $0.12/GB) | Sandbox モードのみ     | 最安値、本番スケールでの実績、初期は Sandbox 解除申請が必要   |

### A. Resend

- API キー 1 本で即日送信可能。Sandbox 申請・審査フロー不要
- `@react-email/components` と公式統合があり、NestJS サービス内から型付きテンプレートを扱える
- 無料枠(3,000 通/月)は MVP フェーズの招待・通知量で十分
- 小規模サービス向け設計のため、月 10 万通クラスのスループットには向かない

### B. Postmark

- 配送性能と Inbox 到達率は 3 候補中最高水準
- $15/月 の固定費が MVP フェーズに対して割高
- Transactional メール特化で、マーケティングメール(将来的に必要になった場合)は別プロダクトを要する

### C. AWS SES

- 大量送信コストは最安。ECS Fargate(本番環境)と同一 AWS アカウントで完結する
- **初期摩擦が大きい**: 新規アカウントは Sandbox モードに置かれ、送信先を AWS コンソールで「認証済みアドレス」として登録しなければならない。一般ユーザーへの招待メール送信には Sandbox 解除申請が必要で、審査完了まで数日かかるケースがある
- Sandbox 制限により MVP のメンバー招待フローが開発初期から動作しない

## 決定

**MVP フェーズは Resend を採用する。本番運用が安定し月間送信数が Resend 有料プランのコスト閾値を超えた段階で AWS SES への移行を検討する。**

`apps/api` には薄い `MailService` 抽象を置き、プロバイダ実装を差し替え可能にする(フォローアップ参照)。

## 理由

### MVP に Resend を選ぶ根拠

- 3 週間 MVP スコープで「メール送信が即日動く」ことが最優先。AWS SES の Sandbox 解除申請は MVP タイムラインのリスクになる
- `@react-email/components` との公式統合により、招待メールのテンプレートを TypeScript で型安全に管理できる。NestJS のサービス層と自然に統合できる
- 無料枠(100 通/日)はメンバー招待・課金通知の初期ボリュームを十分に賄う
- DNS(SPF / DKIM / DMARC)の設定サポートが充実しており、配送性も実用水準

### Postmark を棄却する理由

- 無料枠がテスト用途の 100 通のみで、実 MVP の動作確認に有料契約が必要
- Resend と比較して開発者体験(DX)上の優位性がなく、$15/月 の固定費を正当化しにくい

### SES を現時点で棄却する理由

- Sandbox モード期間中は承認済みアドレスにしか送信できないため、招待メール(不特定ユーザー向け)が本質的に動作しない
- Sandbox 解除申請の審査期間が不定であり、MVP スプリントのブロッカーになるリスクがある
- 本番規模に達した時点(月数万通〜)で改めて選定する価値がある選択肢として保持する

### DNS 要件

Resend を採用した場合でも以下の DNS 設定は必須となる。配送性と「なりすましメール」としてのフィルタリング回避のため、初回セットアップ時に完了させる。

| レコード | 役割                                      |
| -------- | ----------------------------------------- |
| SPF      | 送信元 IP が正規サーバーであることを示す  |
| DKIM     | 送信ドメインの署名検証(Resend が鍵を発行) |
| DMARC    | SPF/DKIM 失敗時のポリシー定義             |

Resend のダッシュボードから CNAME / TXT レコードの値が自動生成されるため、ドメイン DNS に追加するだけで設定が完了する。

## 結果

### 良い影響

- Sandbox 申請なしで MVP 初日から招待メールが動作する
- React Email によりテンプレートを HTML 文字列ではなく TypeScript コンポーネントで管理できる
- `MailService` 抽象を挟むことで、将来の SES 移行がアプリケーションコードの変更なしに完結する
- DNS 設定が Resend ダッシュボードで完結し、SPF / DKIM / DMARC の知識がなくても設定可能

### 悪い影響・リスク

- Resend の無料枠(3,000 通/月)を超えると有料プランへの移行が必要になる
  - 対応: 月次送信数を Resend ダッシュボードで監視し、2,000 通/月 に達した時点で移行計画を立てる
- Resend は新興サービスであり、AWS SES・Postmark に比べて実績・SLA が限られる
  - 対応: 招待メール送信失敗時はアプリ側でエラーハンドリングし、再送 UI を用意する(ユーザーが手動再送できる状態にする)
- 現状の `screen-flow.md` フロー3に「SES 経由」と記載があるが、実装は Resend を使う。図の注記は次の screen-flow.md 更新時に修正する

### フォローアップ

- `apps/api/src/mail/` に `MailService` インターフェースと `ResendMailService` 実装を作成する。プロバイダを差し替える際は `ResendMailService` を `SesMailService` に置き換えるだけで済む設計にする
- 招待メールテンプレートを `packages/mail-templates/`(または `apps/api/src/mail/templates/`) に React Email コンポーネントとして作成する
- shipyard.app ドメインの DNS に Resend 発行の SPF / DKIM / DMARC レコードを設定する(メール送信機能着手前に完了させる)
- 月間送信数が 2,000 通を超えたタイミングで SES 移行の費用対効果を再評価する。移行時は `SesMailService` を実装して差し替える(新 ADR は不要、このフォローアップで意思決定を記録する)
