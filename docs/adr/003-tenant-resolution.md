# ADR-003: テナント解決方式

## ステータス

承認済み(2026-05-01)

## 背景・問題

ユーザーがワークスペースを切り替える際、どの方式で「現在のテナント」を識別するかを決める必要がある。URL設計、認証連携、共有時の挙動などに影響する重要な決定。

関連: ADR-002(マルチテナント分離方式)

## 検討した選択肢

### A. サブドメイン方式

- `acme.shipyard.app` のように各ワークスペースに独自サブドメイン
- 例: Slack、Notion、Zendesk

### B. サブパス方式

- `shipyard.app/w/{slug}` のようにパス単位で識別
- 例: GitHub Organizations、Linear、Vercel Teams

### C. JWT 埋め込み方式

- JWT payload に現在の tenantId を埋め込み、URL は意識しない
- セッション切替で API 呼び分け
- 例: 一部のエンタープライズ SaaS

## 決定

**サブパス方式** を採用する。

## 理由

- ワイルドカード SSL 証明書管理が不要 → 運用シンプル
- DNS 設定不要、Vercel デプロイと相性が良い
- URL でワークスペースが明示される → 共有・ブックマークしやすい
- 誤ったテナントへの遷移が URL で検知可能
- 開発環境(localhost)でも本番同等の挙動を再現できる

棄却理由:

- **サブドメイン方式**: ワイルドカード SSL、DNS 自動化、Cookie ドメイン設計の追加コスト → 3週間 MVP に過大
- **JWT 埋め込み方式**: URL 共有時に「相手側でも自分のテナント」を見せられない、認可ミスの温床

## 結果

### 良い影響

- インフラ構成シンプル、Vercel + ECS で完結
- URL が直感的(`shipyard.app/w/devlog/projects/xxx`)
- スクリーンショット時にテナント context が判別可能
- 招待リンク共有が URL コピーだけで完結

### 悪い影響・リスク

- ブランディング上のサブドメイン優位性が無い
  - 対策: 将来的にカスタムドメイン機能を追加可能で対応(プレミアムプラン要素にできる)
- middleware でパスからの slug 抽出ロジックが必要
  - 対策: Next.js middleware で吸収、API リクエストヘッダーに `X-Tenant-Slug` を付与
- 既存ユーザーが URL を保存している場合の slug 変更の影響
  - 対策: slug 変更時に旧 slug → 新 slug のリダイレクト履歴を保持

### フォローアップ

- Next.js middleware で `/w/{slug}` パターンマッチ実装(Day 4-5)
- API リクエストヘッダーに `X-Tenant-Slug` を付与する規約決定
- slug 命名ルール(英数字+ハイフン、3〜30文字、予約語チェック)を定義
- 将来的なカスタムドメイン対応の設計余地を残す(Tenant テーブルに `customDomain` カラム追加可能な構造)
